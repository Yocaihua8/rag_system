## 1. 项目背景

个人开发者、学习者和知识工作者在本地积累了大量代码项目、文档和笔记，但这些知识分散在文件系统中，难以高效检索、关联和复用。现有工具要么需要联网（隐私风险），要么不支持非结构化知识的语义检索，要么缺乏对本地文件的即时感知。

Knowledge Island（知识岛）旨在成为一个本地优先的个人 AI 第二大脑——将分散的本地知识导入、索引、检索，并通过 AI 问答、掌握度评估帮助用户真正内化和利用自己的知识资产。当前默认交付形态是本地 Web MVP：通过 `backend/app.py` 启动本机 HTTP 服务，在浏览器中完成项目空间、资料导入、检索、问答和评估的完整闭环。

## 2. 项目目标

### 2.1 业务目标

- 成为个人技术知识库的标准化管理工具
- 降低知识检索成本，让用户在 10 秒内找到项目级知识点
- 通过掌握度评估帮助用户发现知识盲区并补强

### 2.2 用户目标

- 快速将本地代码项目、文档、笔记变为可检索知识
- 用自然语言提问并获得有来源支撑的回答
- 持续评估和提升对某一技术领域的掌握程度

### 2.3 技术目标

- 本地优先运行，不强制联网，数据不离开本机
- 支持可选的云端 LLM API（DeepSeek / OpenAI-compatible / Ollama）
- 提供透明的检索质量可观察性，让用户理解回答的可信度
- Web MVP 后端使用 FastAPI + Uvicorn 提供本地 HTTP 服务，`python backend/app.py` 即开即用

## 3. 目标用户

| 用户类型 | 主要诉求 | 典型场景 |
|----------|----------|----------|
| 独立开发者 | 快速查阅项目文档和代码规范 | 面对遗忘的旧代码库时快速定位逻辑和接口 |
| 学习者 | 系统整理和测试对某领域的理解 | 学习 RAG / LLM 后评估自身掌握程度 |
| 知识工作者 | 将分散笔记变成可检索知识库 | 把 Obsidian / Notion 导出内容转为本地问答库 |

## 4. 项目范围

### 4.1 本期纳入范围

- 本地文件夹导入（Markdown、TXT、DOCX、代码、配置文件）
- 单文件上传、文本笔记、URL 摘录（手动粘贴）、剪贴板导入
- RAG 检索（BM25 关键词 + 向量混合，不强制新增检索依赖）
- 自然语言问答（本地 fallback / DeepSeek / OpenAI-compatible / Ollama）
- 多会话聊天记录管理
- 掌握度评估（出题 + 回答评估 + 错题记录）
- 文档集合分组与管理
- 模型 Profile 多配置
- Prompt 预设管理
- 检索调试面板、检索复盘与回答质量反馈
- 项目健康仪表盘
- 备份导出（元数据与聊天记录）
- Docker 一键启动

### 4.2 本期不纳入范围

- 网页自动爬取（B-132，P3 研究项）
- 多用户 / 团队空间（B-118，P3 研究项）
- MCP 插件市场或任意命令执行（B-117，P3 研究项）
- 云端部署 / 多机同步
- 移动端 App
- 语音输入输出
- 图片理解 / 图片生成

旧 PySide6 桌面端代码保留在 `legacy/desktop/` 作为 legacy 参考，不是当前默认入口。

## 5. 成功标准

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 知识库导入成功率 | ≥ 95% | 支持格式的文件成功入库率 |
| 问答有来源回答率 | ≥ 80% | 回答附带有效来源片段比例 |
| 首次完成导入用时 | ≤ 5 分钟 | 从启动到完成第一次知识库导入的时间 |
| 本地冷启动时间 | ≤ 10 秒 | `python backend/app.py` 到浏览器可访问的时间 |

## 6. 约束条件

- **时间约束**：个人兼职项目，无固定发布周期
- **人力约束**：单人开发，需严格控制技术复杂度，避免引入维护成本高的依赖
- **技术约束**：Web MVP 必需运行时依赖保持小而明确（FastAPI + Uvicorn）；可选依赖（如 pymupdf）须明确标注并有降级策略
- **合规约束**：API Key 不得明文持久化；不得收集用户数据发送至外部服务；Agent 工具只允许只读操作
