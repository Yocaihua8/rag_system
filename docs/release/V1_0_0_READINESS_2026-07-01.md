## 1. 发布结论

v1.0.0 发布必须先完成本文的自动化回归、手工主流程冒烟和桌面打包证据收集。任何 P0/P1 阻断项未关闭、必需 CI 未通过、导入/检索/问答/导出/打包主流程任一项不可用时，结论为 **no-go**。

本文只定义发布门禁和验收脚本，不代表 v1.0.0 已发布。实际版本发布时间、Tag 和变更摘要以 `CHANGELOG.md` 与 Git Tag 为准。

## 3. 自动化回归脚本

以下命令按发布前推荐顺序执行。Windows 本机使用 PowerShell；macOS / Linux 可按 CI 命令替换 `.venv\Scripts\python.exe` 为 `.venv/bin/python`。

### 3.1 基线检查

```powershell
git status --short
npm ci
npm audit --audit-level=high
npm run test:unit
npm run build
$env:PYTHONUTF8 = "1"
.venv\Scripts\pip-audit.exe -r requirements.txt -r requirements-dev.txt --progress-spinner off
.venv\Scripts\python.exe -m pytest tests/test_backend tests/test_webapp -q
.venv\Scripts\python.exe scripts\check_docs_consistency.py
```

Go 条件：

- `git status --short` 不包含未解释的发布外改动。
- npm / Python 依赖审计、pytest、Vitest、构建和文档一致性检查退出码为 0。
- 可选依赖降级矩阵通过 `tests/test_backend/test_optional_dependency_matrix.py` 覆盖，缺少 `pymupdf` / `qdrant-client` / `sentence-transformers` 或 Ollama 不可达时不得阻断导入、检索或问答主流程。

### 3.2 主流程 E2E

```powershell
npm run e2e:install
npm run test:e2e
```

如果 Chromium 下载受限，可临时使用本机浏览器：

```powershell
$env:KI_E2E_BROWSER_CHANNEL = "msedge"
npm run test:e2e
```

Go 条件：

- Playwright 用隔离 `KI_DB_PATH` 启动后端，不写入默认 `runtime/webapp/knowledge_island.db`。
- E2E 完成项目创建、文本笔记导入和工作台问答，回答区包含来源。

### 3.3 导入 / 检索 / 问答 / 导出重点回归

```powershell
.venv\Scripts\python.exe -m pytest `
  tests/test_webapp/test_api.py `
  tests/test_webapp/test_search.py `
  tests/test_webapp/test_document_processing.py `
  tests/test_webapp/test_chat_history.py `
  tests/test_webapp/test_agent_tools.py `
  tests/test_webapp/test_vector_store_sync.py `
  -q
```

其中结果导出必须覆盖：

- Markdown：`tests/test_webapp/test_api.py::test_result_export_api_writes_markdown_file_for_chat_message`
- PDF：`tests/test_webapp/test_api.py::test_result_export_api_writes_pdf_file_for_chat_message`
- 非法格式与跨项目消息拒绝：`tests/test_webapp/test_api.py::test_result_export_api_rejects_invalid_format_and_cross_project_message`

### 3.4 桌面打包预检

```powershell
.venv\Scripts\python.exe -m pytest tests/test_webapp/test_tauri_packaging.py -q
npx tauri info
npm run tauri:build:windows
```

Go 条件：

- `test_tauri_packaging.py` 通过，覆盖 Tauri 配置、sidecar 脚本、图标清单和 workflow。
- `npx tauri info` 不显示会阻断当前目标平台构建的缺失项。
- 发布 Windows 桌面包时，`npm run tauri:build:windows` 生成 NSIS installer。

macOS / Linux 原生包必须在目标平台或手动 GitHub Actions workflow 验证：

```powershell
gh workflow run "Tauri Packaging"
gh run list --workflow "Tauri Packaging" --limit 5
gh run watch <run-id> --exit-status
```

Go 条件：

- workflow 生成 `knowledge-island-macos-dmg` 与 `knowledge-island-linux-appimage` artifact。
- artifact 分别包含 `.dmg` 与 `.AppImage`，且 upload step 未出现 `if-no-files-found` 错误。

## 4. 手工主流程冒烟

手工冒烟建议使用临时数据库和输出目录，避免污染本机默认数据：

```powershell
New-Item -ItemType Directory -Force tmp\release-v1.0.0 | Out-Null
$env:KI_DB_PATH = "$PWD\tmp\release-v1.0.0\knowledge_island.db"
$env:KI_OUTPUT_DIR = "$PWD\tmp\release-v1.0.0\outputs"
.venv\Scripts\python.exe app.py
```

浏览器打开 `http://127.0.0.1:8765` 后执行：

1. 创建项目空间，绑定一个临时目录。
2. 在临时目录放入 Markdown 或 TXT 文件，执行“同步当前项目目录”，确认导入统计包含新增文档。
3. 进入资料库，确认文档列表、文件预览、导入批次历史可见。
4. 在工作台搜索该文档中的关键词，确认命中结果包含路径、片段和分数信息。
5. 提问一个只能由该文档回答的问题，确认回答包含来源；未配置模型时允许本地片段组合回答。
6. 通过 `POST /api/export/result` 对同一条回答执行结果导出，分别验证 Markdown 与 PDF 文件写入 `KI_OUTPUT_DIR`；当前不要求前端提供导出按钮。
7. 切换/刷新页面，确认聊天记录可恢复，来源仍可查看。
8. 触发一个负向路径：空项目搜索或删除项目目录后导入，确认错误提示可读且不写入假数据。

## 6. No-Go 处理

出现 no-go 时：

1. 不打 `v1.0.0` Tag，不发布安装包或 release note。
2. 在 `docs/BACKLOG.md § 5` 新增或更新对应阻断项，优先级按影响定为 P0/P1。
3. 保留失败命令、日志摘要和 artifact 链接，避免只写“验证失败”。
4. 修复后重新执行本文 § 3 和 § 4；不得只复测单个失败命令就改判 Go。
