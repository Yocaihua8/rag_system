# 项目知识库助手设计

> 状态：Draft
> 日期：2026-05-11
> 范围：产品定位、主界面信息架构、项目导入、知识点提炼、掌握评估、知识库辅助管理
> 约束：本文是设计文档，不代表现行代码已经实现

## 1. 结论

项目应从以简历生成为主的 Career Assistant 调整为“项目知识库助手”。界面上不使用“第二大脑”这类概念化词语，而使用更直观的“项目问答 / 我的项目 / 知识库 / 掌握评估 / 设置”。

第一版目标是形成一个可验证闭环：

```text
导入代码项目
→ 建立项目知识库
→ 提炼项目知识点
→ 围绕项目问答
→ 自动生成评估题
→ 用户回答
→ 输出掌握情况与能力差距
```

简历、JD 匹配、面试脚本保留为后续输出工具，不作为第一版主线。

## 2. 目标与非目标

### 2.1 第一版目标

- 让用户通过选择代码项目目录完成导入。
- 自动扫描 README、docs、源码、配置文件和测试文件。
- 建立可问答的本地 RAG 索引。
- 从项目内容中提炼技术栈、模块职责、核心流程、配置、测试和难点。
- 支持 Codex 式项目问答，答案展示来源文件和片段。
- 支持“开始评估”，由系统围绕项目自动出题。
- 根据用户回答输出掌握情况、漏掉的关键点、建议阅读文件和下一步学习顺序。
- 提供类似 SAS 后台式知识库的辅助管理页，用于追踪入库状态、文件、知识点和题库。

### 2.2 第一版不做

- 不拆 Web 前端，继续使用现有 PySide6 桌面端。
- 不做复杂知识图谱编辑器。
- 不做自动学习排期。
- 不让 AI 自动修改正式知识结构。
- 不把简历生成、JD 匹配、面试脚本放在主导航核心位置。
- 不优先支持 PDF/DOCX 等重解析场景，先以代码项目与 Markdown/TXT/源码文件为主。

## 4. 主界面布局

第一版采用“Codex 聊天式主体验 + SAS 后台式辅助管理”的组合。

```text
┌──────────────┬─────────────────────────────┬──────────────────┐
│ 左侧项目列表 │ 中间项目问答 / 掌握评估       │ 右侧依据与差距     │
├──────────────┼─────────────────────────────┼──────────────────┤
│ 导入项目     │ 输入问题                     │ 引用来源           │
│ 当前项目     │ 查看回答                     │ 相关文件           │
│ 索引状态     │ 开始评估                     │ 项目知识点         │
│ 掌握情况摘要 │ 回答评估题                   │ 掌握情况           │
│              │ 继续追问                     │ 下一步建议         │
└──────────────┴─────────────────────────────┴──────────────────┘
```

用户日常主要在“项目问答”里操作。遇到入库失败、知识点不完整、评估题来源不清晰时，再进入“知识库”页排查。

## 5. 首次使用流程

首次启动弹窗文案应保持直接：

```text
欢迎使用项目知识库

1. 导入项目
选择一个代码项目目录，系统会读取 README、docs、源码和配置文件。

2. 建立索引
系统会整理项目文件，提炼项目知识点，生成可问答的知识库。

3. 开始使用
你可以围绕项目提问，也可以开始评估，了解自己对项目的掌握情况。
```

按钮：

```text
先看看界面
导入第一个项目
```

## 6. 项目导入流程

```text
用户选择代码项目目录
→ 扫描文件
→ 过滤无关目录
→ 保存文件记录
→ 分块
→ 建立索引
→ 提炼项目知识点
→ 更新项目状态
```

默认过滤目录：

```text
.git
node_modules
.venv
dist
build
__pycache__
.pytest_cache
.idea
```

项目状态对外文案：

| 内部状态 | 界面文案 | 说明 |
|----------|----------|------|
| pending | 待处理 | 已创建项目，尚未开始 |
| running | 处理中 | 正在扫描、索引或提炼 |
| success | 可使用 | 项目可问答和评估 |
| blocked | 需确认 | 部分文件或知识点需要用户确认 |
| failed | 失败 | 处理失败，需要查看原因 |

## 7. 项目知识点提炼

第一版采用“规则优先 + AI 增强”的保守策略，避免全部依赖模型。

规则来源：

- README 标题与章节。
- docs 目录标题与正文。
- 文件路径和目录结构。
- 配置文件，如 `requirements.txt`、`package.json`、`pyproject.toml`、`docker-compose.yml`。
- 测试目录和测试文件名。
- 技术关键词。

知识点类型：

| 类型 | 说明 |
|------|------|
| tech_stack | 技术栈 |
| module | 模块职责 |
| flow | 核心流程 |
| config | 配置与启动方式 |
| test | 测试方式 |
| concept | 关键概念 |
| risk | 难点或风险 |

每个知识点必须保留来源文件或来源片段，避免形成不可追踪的 AI 结论。

## 8. 掌握评估流程

用户在“项目问答”或“掌握评估”页点击“开始评估”：

```text
选择项目
→ 选择范围：全部 / 架构 / 后端 / 前端 / 测试 / 配置
→ 系统生成 5-10 道题
→ 用户逐题回答
→ 系统检索相关来源
→ 生成参考要点
→ 对照用户回答
→ 输出单题反馈
→ 汇总能力差距
```

第一版题型：

| 题型 | 示例 |
|------|------|
| 概念理解题 | 这个模块主要解决什么问题？ |
| 流程说明题 | 从导入项目到完成问答经过哪些步骤？ |
| 代码定位题 | 这个功能主要在哪些文件里实现？ |

第一版不做选择题。原因是选择题更偏记忆测试，不能稳定判断用户是否真正理解项目。

## 9. 评估结果

单题结果使用学习型表达：

```text
已掌握
基本理解
需要补充
暂未掌握
```

每题反馈包含：

- 判断结果。
- 判断原因。
- 用户回答中覆盖的关键点。
- 用户漏掉的关键点。
- 建议阅读的文件。
- 可继续追问的问题。

能力差距报告包含：

- 掌握概览。
- 已掌握知识点。
- 需要补充的知识点。
- 建议阅读文件。
- 建议追问问题。
- 下一步学习顺序。

## 10. 知识库辅助管理页

知识库页参考 SAS 后台式知识库，但保持轻量，只做四块：

```text
项目入库状态
文件列表
项目知识点
评估题库
```

### 10.1 项目入库状态

展示：

- 当前项目是否可问答。
- 上次索引时间。
- 文件数量。
- 知识点数量。
- 题目数量。
- 最近错误或需确认原因。

### 10.2 文件列表

展示：

- 文件路径。
- 文件类型。
- 是否已索引。
- 分块数量。
- 是否参与知识点提炼。

### 10.3 项目知识点

展示：

- 知识点名称。
- 类型。
- 来源文件。
- 是否用于评估。
- 最近评估结果。

### 10.4 评估题库

展示：

- 题目。
- 题型。
- 绑定知识点。
- 来源文件。
- 最近回答结果。

## 11. 架构与模块拆分

保持现有六边形架构，新增能力放在 application / domain / ports / adapters 中。

建议新增领域模型：

```text
ProjectKnowledgePoint
AssessmentQuestion
AssessmentAnswer
AssessmentResult
AbilityGapReport
```

建议新增端口：

```text
IKnowledgePointStore
IAssessmentStore
```

建议新增 SQLite 适配器：

```text
SqliteKnowledgePointStore
SqliteAssessmentStore
```

建议新增用例：

```text
ProjectAnalysisUseCase
AssessmentUseCase
```

职责：

| 模块 | 职责 |
|------|------|
| ProjectAnalysisUseCase | 分析项目结构，提炼项目知识点 |
| AssessmentUseCase | 生成评估题、提交回答、评估回答、生成报告 |
| IKnowledgePointStore | 保存和查询项目知识点 |
| IAssessmentStore | 保存题目、回答、结果和报告 |

## 12. 分阶段落地

### 阶段 1：文档与产品定位

- 调整 README 和架构文档中的产品定位。
- 在 BACKLOG 中新增项目知识库与掌握评估任务队列。
- 在 DEVLOG 记录本次设计决策。

### 阶段 2：界面骨架

- 主导航改为“项目问答 / 我的项目 / 知识库 / 掌握评估 / 设置”。
- 首次使用弹窗改为三步项目导入流程。
- 新增掌握评估页面占位。
- 简历生成降级为后续输出工具，不占主导航核心位置。

### 阶段 3：项目知识点模型

- 新增项目知识点模型、端口、SQLite 存储和测试。
- 在索引完成后写入可追踪的知识点。

### 阶段 4：项目知识点提炼

- 先用规则提炼 README、docs、路径、配置和测试信息。
- 再用 LLM 对规则结果做补充说明。
- 每个知识点必须保留来源。

### 阶段 5：自动出题

- 基于知识点生成 5-10 道评估题。
- 每道题绑定知识点和来源。
- 第一版只做概念理解、流程说明、代码定位三类题。

### 阶段 6：回答评估与能力差距

- 检索相关来源。
- 生成参考要点。
- 对照用户回答给出等级和原因。
- 汇总项目能力差距报告。

### 阶段 7：知识库辅助管理

- 展示项目状态、文件列表、项目知识点、评估题库和最近评估结果。
- 用于排查“为什么问答没引用某文件 / 为什么没生成某类题 / 为什么某题判定不掌握”。

## 13. 测试关注点

- 项目扫描默认过滤 `.git / node_modules / .venv / dist / build` 等目录。
- 项目状态能正确区分待处理、处理中、可使用、需确认、失败。
- 知识点提炼必须带来源文件。
- 自动出题必须绑定知识点和来源。
- 空知识点时不能生成无依据题目。
- 回答评估不能在无来源时给出确定性判断。
- 能力差距报告必须包含建议阅读文件。
- PySide6 UI 中主导航、首次弹窗、空状态文案不出现“第二大脑”。
- 现有问答、索引、设置测试不应回归。

## 14. 风险说明

| 风险 | 说明 | 缓解 |
|------|------|------|
| AI 评估不可信 | 用户回答可能被模型误判 | 每题必须展示来源、参考要点和判断原因 |
| 项目文件过多 | 代码仓库可能包含依赖、构建产物和缓存 | 默认过滤常见无关目录，并显示文件统计 |
| 文案概念化 | “第二大脑”对新用户不直观 | UI 统一使用项目知识库、项目问答、掌握评估 |
| 知识点不可追踪 | AI 直接生成结论会形成黑盒 | 每个知识点、题目、评估结果都绑定来源 |
| 范围膨胀 | 同时做图谱、Web、排期、简历会拖慢第一版 | 第一版只做项目问答、知识点、评估和差距报告 |
