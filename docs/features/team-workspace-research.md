# 多用户 / 团队空间研究

> 状态：Active
> Owner：RAG 团队
> Last Updated：2026-06-30
> Scope：B-118 多用户 / 团队空间研究结论；本文件不代表已实现多用户或团队空间
> Related：docs/design/permission-matrix.md, docs/requirements/project-background-and-scope.md, docs/requirements/functional-modules.md, docs/design/system-design-overview.md, docs/BACKLOG.md

## 1. 研究结论

B-118 结论：Knowledge Island 当前继续保持**本地单用户应用**定位，不在当前版本实现多用户、团队空间、组织、成员邀请、角色权限或共享知识库。

现有 B-140 认证中间件只是单用户远程访问保护：

- 只有一个隐式"本机用户"。
- `X-API-Key` / Bearer JWT 代表同一个本机用户的访问凭证。
- 没有 `users`、`teams`、`memberships`、`roles`、`permissions` 或 `owner_id` 数据模型。
- 当前 `project_id` 只隔离本机用户的不同项目空间，不是租户边界。

因此，B-118 只关闭研究项；不修改运行时代码、不新增 API、不新增数据库 schema。

## 2. 当前基线

| 领域 | 当前状态 |
|------|----------|
| 认证 | 默认关闭；启用后使用单个管理 API Key 或短期 JWT |
| 用户模型 | 无用户表，无注册/登录页，无账号身份 |
| 项目隔离 | 数据按 `project_id` 过滤，用于同一用户的项目空间隔离 |
| 权限模型 | 无 RBAC；本机用户对所有项目有完整读写权限 |
| 数据库 | 大多数业务表已有 `project_id`，但没有 `user_id`、`team_id`、成员关系或所有者字段 |
| 前端 | 没有组织切换、成员管理、邀请、角色展示或共享设置 |
| 审计 | 仅 Agent 工具运行、导入批次、检索复盘等业务审计；没有用户行为审计 |

## 3. 能力差距

真正的团队空间不是在现有项目表上加一个名称字段即可完成。至少需要补齐以下能力：

| 能力 | 当前缺口 | 为什么不能直接实现 |
|------|----------|-------------------|
| 身份体系 | 没有账号、密码、OIDC、邮箱或设备身份 | 无法区分操作者，无法做成员权限和审计 |
| 租户模型 | 没有 team / organization / workspace owner | `project_id` 只能隔离项目，不能表达团队归属 |
| 成员关系 | 没有 membership、邀请、退出、禁用 | 无法控制谁能访问团队空间 |
| RBAC | 没有 owner/admin/editor/viewer 等角色 | 当前 API 默认同一用户全权限 |
| 数据授权 | 路由只校验项目存在和 project_id 所属关系 | 需要在每个读取/写入接口上增加身份与权限校验 |
| 密钥隔离 | 模型 Profile 只保存本机 Key 引用 | 团队共享时需定义谁能配置、谁能使用、谁能查看状态 |
| 文件访问 | 本地目录访问依赖运行进程的 OS 权限 | 团队成员远程访问时不能默认读取服务端本机任意目录 |
| 审计合规 | 只有少量业务运行记录 | 团队场景需要成员级行为审计和敏感操作追踪 |

## 4. 推荐路线

不建议在当前 Web MVP 直接实现完整团队空间。若未来要进入实现，建议拆成独立阶段：

### 阶段 A：单用户身份基线

- 把 B-140 从"单个管理 Key"升级为明确的单用户账号或本机管理员身份。
- 明确 token 生命周期、退出登录、刷新策略和本地安全存储。
- 继续不引入团队，只为后续多用户打基础。

### 阶段 B：租户与项目所有权

- 新增 `users`、`teams`、`team_memberships` 等核心模型。
- 为 `projects` 增加所有者或团队归属。
- 制定从现有单用户 SQLite 数据迁移到默认个人空间的方案。

### 阶段 C：RBAC 与接口授权

- 定义角色：owner / admin / editor / viewer。
- 为每个 API 明确最低权限，例如导入需要 editor，删除项目需要 owner/admin，问答需要 viewer。
- 在路由层统一执行授权，禁止业务层或前端绕过。

### 阶段 D：团队协作体验

- 增加团队切换、成员管理、邀请、共享项目列表和只读成员体验。
- 增加成员级操作审计。
- 明确团队空间内模型 Profile、Prompt 预设、文档集合、聊天记录和导出权限。

每个阶段都应作为单独 BACKLOG 项和 ADR 处理，不应由 B-118 直接展开实现。

## 5. 非目标

B-118 后续不应直接包含：

- 用户注册、密码登录、OAuth/OIDC 或邮箱邀请。
- 团队/组织表、成员表、角色表或权限表。
- 多租户数据库迁移。
- 共享项目空间、多人同时编辑、评论、消息通知。
- 云端部署、多机同步、公开分享链接。
- 前端团队管理页面。
- 把现有单用户认证强行解释为多用户登录。

## 6. 安全准入清单

未来任何多用户 / 团队空间实现任务开始前，必须先明确：

- **身份来源**：本地账号、OIDC、第三方登录或单机管理员账户。
- **租户边界**：个人空间、团队空间和项目空间的层级关系。
- **授权矩阵**：每个 API 的最低角色要求。
- **数据归属**：项目、文档、chunk、vector、聊天、Prompt、模型 Profile、导出文件属于谁。
- **密钥策略**：团队场景下 API Key 由谁配置、谁可使用、是否可继承、是否可撤销。
- **迁移策略**：现有单用户 SQLite 数据如何迁移为默认个人空间。
- **审计策略**：成员邀请、删除项目、导入、导出、模型配置、工具运行等敏感操作如何记录。
- **部署边界**：仍是本机局域访问，还是开始支持远程团队服务；两者安全模型不同。

## 7. 验收建议

若未来新建实现任务，最低测试应覆盖：

- 认证关闭时仍保持当前本地单用户体验。
- 认证启用时，未授权用户不能读取任何项目列表、文档、聊天或导出内容。
- viewer 只能读取和问答，不能导入、删除、修改模型配置或导出敏感数据。
- editor 可导入和管理文档，但不能删除团队或管理成员。
- admin 可管理成员和团队配置，但敏感密钥不回显。
- owner 才能删除团队空间或转移所有权。
- 跨 team 的 `project_id`、`document_id`、`chat_message_id`、`tool_run_id` 均被拒绝。
- 迁移后的历史项目默认归属个人空间，不暴露给其他成员。

## 8. 当前状态声明

截至 B-118 完成时，Knowledge Island 的正式状态仍是：

- 本地单用户应用。
- B-140 提供可选单用户认证，不是账号体系。
- 项目空间是单用户内的知识域隔离，不是团队租户。
- 多用户 / 团队空间仅有研究结论，没有运行时代码或 API 承诺。
