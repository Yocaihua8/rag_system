# DevLog 开发日志

> 状态：Active
> Owner：RAG 团队
> Last Updated：2026-05-25

用于记录**开发过程中的日常进展、遇到的问题、临时决策与下一步动作**。是面向开发者自己和同伴的"过程记录"，不是面向用户的版本说明。

## 1. 定位与边界

| 文档 | 面向 | 内容 | 节奏 |
|------|------|------|------|
| `../../CHANGELOG.md` | 使用者 / 维护者 | 已发布版本的对外变更 | 按版本 |
| `../BACKLOG.md` | 团队 | 待办、已知问题、技术债 | 随时更新 |
| `../adr/*.md` | 团队 | 重大/定稿的架构决策 | 按决策点 |
| `./*.md` | 开发者自己 | 当天/当周做了什么、卡在哪、下一步 | 日 / 周 |

**一句话区分**：CHANGELOG 是"发给别人看的"，BACKLOG 是"要做什么"，ADR 是"为什么这么定"，**DevLog 是"今天/这周做到哪里了"**。

## 2. 使用规则

- 必须：每条 devlog 只写**过程性事实**，不写长篇设计；设计定稿请写入 `../design/` 或 `../adr/`
- 必须：跨越 devlog 的**决策**必须提升到 ADR，并在 devlog 里引用 ADR 编号
- 应该：发现的**问题**同步进 `BACKLOG.md`，devlog 只留指针
- 应该：日志长度控制在 150 行以内，太长说明应拆分或提升到正式文档
- 可选：个人项目可用日报；团队项目推荐周报

## 3. 文件命名

- 日报：`YYYY-MM-DD.md`，例如 `2026-04-17.md`
- 周报：`YYYY-Www.md`，例如 `2026-W16.md`
- 同一天多条：追加到当天文件末尾，不新建文件

## 4. 写作建议

- 先写"做了什么"，再写"卡在哪"，最后写"下一步"
- 用最短的句子，不要写成散文
- 贴链接优先于重复描述：引用 commit、PR、BACKLOG 条目 ID
- 不要把 devlog 当调试草稿纸——最终结论请回流到正式文档

## 5. 模板

新建条目时复制 `devlog-template.md`，按模板字段填写。
