# 文档写作规范

> 状态：Active
> Owner：RAG 团队
> Last Updated：2026-05-16

本文档定义本仓库文档的统一写作规则。目标是避免“已实现内容”和“规划内容”混写，以及不同文档使用不同语气导致的维护成本。

## 1. 基本原则

- 文档先结论：先写“当前结论/边界”，再写细节实现。
- 一文一主题：一篇文档只承担一个职责。
- 已落地与规划分离：定稿文档只写已落地行为，待做事项写 `BACKLOG.md` 或 `ADR`。
- 标题与术语统一：一个语义只保留一个名字（本项目使用“项目空间”而非“工作区”对外文案）。

## 2. 文档头部模板（除 `CHANGELOG.md` 外）

```text
# 标题

> 状态：Draft / In Review / Active / Deprecated / Archived
> Owner：RAG 团队
> Last Updated：YYYY-MM-DD
> Scope：可选，按文档粒度填写
> Related：可选，相关文档路径
```

## 3. 章节结构

推荐顺序：

1. 目的 / 背景
2. 范围 / 边界
3. 结论 / 规则
4. 行为细节
5. 示例 / 附录（如有）

## 4. 命名与样式

- 文件名使用小写 `kebab-case`。
- 同一文档中优先使用短句，避免重复否定。
- 中文文档使用全角中文标点，代码、命令、变量名保留半角。
- 状态词汇统一使用：
  - **必须**：不允许偏离
  - **应该**：建议执行，可在文内说明例外
  - **可选**：按场景实施
  - **禁止**：明确不应出现

## 5. 链接与图示

- 文档内交叉链接优先使用相对路径（如 `design/architecture-overview.md`）。
- 外部链接带完整协议，尽量附来源日期。
- 页面结构和架构关系尽量用文本图 + 表格，复杂关系可用 Mermaid。

## 6. 文档维护

- 文档行为改动必须同步：
  - `requirements/*`（已落地边界）
  - `design/*`（架构与数据）
  - `guides/testing.md`（验证方式）
  - `BACKLOG.md`（未完事项）
  - `CHANGELOG.md`（对外变更）
- `docs/README.md` 的目录说明与实际文件必须保持一致。
